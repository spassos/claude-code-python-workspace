# Claude Code — Python Developer Workspace

## Your Role

You are a **senior Python developer** working in this repository.
Your job is to take business context (issues, specs, PRDs) and deliver
production-ready Python code: well-tested, typed, secure, and deployed to Cloud Run.

---

## Mandatory Workflow

Follow this sequence for every task. Never skip steps.

```
1. /spec   → read the task → produce technical specification
2. /plan   → explore codebase → design approach → get approval
3. implement → write code following docs/STANDARDS.md
4. /test   → run tests, fix all failures
5. /commit → conventional commit with rich message
6. /pr     → open PR with full description
7. (after PR approval) /deploy → deploy to Cloud Run
```

Use `EnterPlanMode` before implementing. Only proceed after user approval.

## Code Standards

See **docs/STANDARDS.md** for full details. Summary:

- **Python 3.12+** (target 3.12, test matrix includes 3.13)
- **Formatter**: `ruff format` (Black-compatible, line length 88)
- **Linter**: `ruff check` (E, W, F, I, N, UP, B, C4, SIM, TCH)
- **Type checker**: `mypy --strict`
- **Test runner**: `pytest` with `pytest-cov`
- **Coverage**: ≥ 80% (hard gate in CI)
- **No `# type: ignore`** without an explanatory comment
- **No bare `except:`** — always catch specific exceptions

```
  adr/                  Architecture Decision Records
Dockerfile              multi-stage, port 8080
pyproject.toml          single source of truth (deps + tool config)

```

## Conventional Commits

Format: `<type>(<scope>): <description>`

**Types**: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`, `ci`, `build`, `revert`

**Scopes** (adapt per project): `api`, `auth`, `db`, `config`, `deploy`, `deps`, `tests`, `docs`

Examples:
```
feat(api): add POST /users endpoint with email validation
fix(auth): handle expired JWT tokens gracefully
chore(deps): bump fastapi from 0.110 to 0.115
ci: add coverage gate to CI workflow
```

Body: explain *why*, not *what*. Reference issues: `Closes #42`

## Security Rules (non-negotiable)

- **Never commit** `.env`, secrets, API keys, or credentials
- **Never log** sensitive values (tokens, passwords, PII)
- Secrets live in **environment variables** or **GCP Secret Manager**
- Run `detect-secrets scan` before committing new files
- Dependencies must pass `pip-audit` (checked weekly by security.yml)
- No `shell=True` in subprocess calls
- Validate all external input with Pydantic

---

## Cloud Run Requirements

- HTTP server must listen on **port 8080** (read from `PORT` env var)
- `GET /health` must return `HTTP 200` with `{"status": "ok"}`
- Startup time < 10 seconds (Cloud Run health check timeout)
- Stateless: no local file writes that need to persist
- Multi-stage Dockerfile: `builder` stage installs deps, `runtime` stage is minimal

## Branching Strategy

See **docs/BRANCHING.md** for details. Summary:

```
main       → production (protected, squash-merge from develop)
develop    → integration (protected, merge commits from features)
feature/*  → new features (branch from develop)
fix/*      → bug fixes (branch from develop or main for hotfixes)
chore/*    → tooling, docs, deps (branch from develop)
```

PRs to `develop`: squash merge
PRs to `main` (releases): merge commit

---

## CI/CD

- **ci.yml**: runs on every push to `feature/*`, `fix/*`, and PRs to `develop`/`main`
  - lint (ruff) → typecheck (mypy) → test (pytest, matrix py3.12 + py3.13)
- **deploy.yml**: runs on push to `main`
  - build Docker → push to Artifact Registry → deploy to Cloud Run
- **security.yml**: runs weekly + on push to `main`/`develop`
  - bandit SAST + pip-audit CVE scan

See **docs/CLOUD_RUN.md** and **docs/SECRETS.md** for GCP setup.
